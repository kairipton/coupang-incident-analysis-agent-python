# 서버 운영 표준 절차서 (SOP) - 간소화 버전
**문서 등급:** 대외비
**시스템 모드:** 텍스트 기반 상태 관제 (Text-based Monitoring)

---

## 1. CPU 상태 관리 (CPU Status)

### SOP-01: CPU 상태 - 과부하 (CPU_OVERLOAD)
* **감지 상태:** `CPU: OVERLOAD`
* **원인:** 특정 프로세스(PID)가 무한 루프에 빠짐.
* **증상:** 시스템 반응이 매우 느려지며, 특정 PID가 자원을 독점함.
* **해결 절차:**
    1. 문제가 되는 프로세스 ID(PID)를 찾는다. (주로 로그에 표시됨)
    2. `kill_process(pid)`를 실행하여 강제 종료한다.

### SOP-02: CPU 상태 - 멈춤 (CPU_STUCK)
* **감지 상태:** `CPU: DEADLOCK`
* **원인:** 서비스 간의 자원 경합(Deadlock) 발생.
* **증상:** CPU 부하는 심하지 않으나, 서비스가 동작하지 않고 멈춰있음.
* **해결 절차:**
    1. 프로세스 종료로는 해결되지 않음.
    2. `restart_service(service_name)`을 실행하여 서비스를 재시작한다.

## 2. 메모리 상태 관리 (Memory Status)

### SOP-03: 메모리 상태 - 누수 (MEMORY_LEAK)
* **감지 상태:** `MEMORY: LEAK`
* **원인:** 애플리케이션 캐시 데이터가 해제되지 않고 계속 쌓임.
* **증상:** 가비지 컬렉션(GC) 오류 메시지 발생.
* **해결 절차:**
    1. 서버가 다운되기 전에 공간을 확보해야 함.
    2. `clear_cache()`를 실행하여 임시 메모리를 비운다.

### SOP-04: 디스크 상태 - 가득 참 (DISK_FULL)
* **감지 상태:** `MEMORY: DISK_FULL`
* **원인:** 로그 파일 용량이 너무 커서 저장 공간 부족.
* **증상:** 메모리 누수와 비슷해 보이지만, 원인이 '파일 용량'임.
* **해결 절차:**
    1. 캐시 삭제는 효과 없음.
    2. `rotate_logs()`를 실행하여 로그 파일을 압축/정리한다.

## 3. 네트워크 상태 관리 (Network Status)

### SOP-05: 네트워크 상태 - 공격 감지 (NETWORK_DDOS)
* **감지 상태:** `NETWORK: ATTACK`
* **원인:** 외부 IP에서의 대량 트래픽 유입 공격.
* **증상:** 특정 IP 주소(예: 192.168...)가 로그에 반복적으로 나타남.
* **해결 절차:**
    1. 공격자 IP를 식별한다.
    2. `block_ip(ip_address)`를 실행하여 차단한다.
    * **주의:** 반드시 관리자 승인(Approval)을 받을 것.

### SOP-06: 데이터베이스 상태 - 연결 실패 (DB_FAIL)
* **감지 상태:** `DB: DISCONNECT`
* **원인:** DB 연결 풀(Pool)이 꽉 차서 새로운 연결 불가.
* **증상:** 네트워크 공격은 없으나 DB 응답이 없음.
* **해결 절차:**
    1. IP 차단은 해결책이 아님.
    2. `reset_connection_pool()`을 실행하여 연결을 초기화한다.
